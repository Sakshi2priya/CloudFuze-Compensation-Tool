# Migration Guide: MySQL → PostgreSQL

This document describes how to switch the Compensation Tool from **MySQL** to **PostgreSQL** and what to change in the code. **No code has been changed yet**—use this as a checklist when you are ready to migrate.

---

## 1. Environment & Setup

### 1.1 Install PostgreSQL
- Install PostgreSQL (e.g. from https://www.postgresql.org/download/).
- Default port is **5432** (MySQL uses 3306).
- Create a database, e.g. `cloudfuze_incentive`.

### 1.2 Environment variables
Keep the same names; only the default port changes:

| Variable           | MySQL (current) | PostgreSQL          |
|--------------------|-----------------|---------------------|
| `COMP_DB_HOST`     | localhost       | localhost           |
| `COMP_DB_PORT`     | 3306            | **5432**            |
| `COMP_DB_NAME`     | your_db         | your_db             |
| `COMP_DB_USER`     | your_user       | your_user           |
| `COMP_DB_PASSWORD` | your_password   | your_password       |

---

## 2. Dependencies

### 2.1 `requirements.txt`
- **Remove:** `mysql-connector-python>=9.0.0`
- **Add:** `psycopg2-binary>=2.9.0` (or `psycopg2` if you install libpq yourself)

Then run: `pip install -r requirements.txt`

## 3. Code Changes in `database.py`

### 3.1 Imports (top of file)
- **Remove:**
  ```python
  import mysql.connector
  from mysql.connector.connection import MySQLConnection
  ```
- **Add:**
  ```python
  import psycopg2
  from psycopg2.extras import RealDictCursor
  ```

### 3.2 `get_db_config()`
- Change **default port** from `3306` to `5432`.
- **Optional:** psycopg2 uses the same keys (`host`, `port`, `database`, `user`, `password`), so the returned dict can stay as is. Ensure `database` and `port` are correct for PostgreSQL.

### 3.3 `get_connection()`
- **Replace** `mysql.connector.connect(**get_db_config())` with:
  ```python
  return psycopg2.connect(**get_db_config())
  ```
- **Remove** the return type hint `MySQLConnection` (use something like `psycopg2.extensions.connection` or omit).

### 3.4 `db_cursor()`
- MySQL uses `cursor(dictionary=True)` for dict-like rows. In PostgreSQL with psycopg2 you must use **RealDictCursor**:
  ```python
  cursor = conn.cursor(cursor_factory=RealDictCursor)
  ```
  So the line `cursor = conn.cursor(dictionary=True)` becomes:
  ```python
  cursor = conn.cursor(cursor_factory=RealDictCursor)
  ```

### 3.5 Exception handling
- **Replace** all `except mysql.connector.Error as e` with:
  ```python
  except psycopg2.Error as e:
  ```
  Occurrences are in:
  - `initialize_schema()` (multiple blocks)
  - `_ensure_deals_extra_columns()` (around line 501)
  - `_ensure_rep_incentives_extra_columns()` (around line 611)
- Error message checks like `"Duplicate column" not in str(e)` can stay; PostgreSQL uses different messages (e.g. `"already exists"`), so you may want to also allow `"already exists"` in the condition.

### 3.6 `cursor.lastrowid` → PostgreSQL
PostgreSQL does not have `lastrowid`. Use **RETURNING** in the INSERT and fetch the id.

- **`create_user()`** (returns `user_id`):
  - Change INSERT to:
    ```sql
    INSERT INTO users (full_name, email, password_hash, role, team_id)
    VALUES (%s, %s, %s, %s, %s)
    RETURNING user_id
    ```
  - After `cursor.execute(...)`, use: `return int(cursor.fetchone()["user_id"])` instead of `return int(cursor.lastrowid)`.

- **`create_upload()`** (returns `upload_id`):
  - Add `RETURNING upload_id` to the INSERT.
  - Use `return int(cursor.fetchone()["upload_id"])` instead of `cursor.lastrowid`.

- **`insert_legacy_partner_compensation()`** (around line 851, returns `id`):
  - Add `RETURNING id` to the INSERT into `partner_compensation`.
  - Use `return int(cursor.fetchone()["id"])` instead of `cursor.lastrowid`.

- **Placeholder:** psycopg2 uses `%s` for parameters (same as your current code), so no change there.

### 3.7 `INSERT IGNORE` → PostgreSQL
- **Location:** `_seed_default_data()` — `INSERT IGNORE INTO teams (team_name) VALUES (%s)`.
- **Replace with:**
  ```sql
  INSERT INTO teams (team_name) VALUES (%s)
  ON CONFLICT (team_name) DO NOTHING
  ```
  This requires `team_name` to have a UNIQUE constraint (you already have it).

### 3.8 Schema (CREATE TABLE and ALTER) — MySQL vs PostgreSQL

| MySQL                          | PostgreSQL equivalent |
|--------------------------------|------------------------|
| `INT AUTO_INCREMENT PRIMARY KEY` | `SERIAL PRIMARY KEY` or `GENERATED BY DEFAULT AS IDENTITY` |
| `TINYINT(1)`                   | `SMALLINT` or `BOOLEAN` (use `TRUE`/`FALSE`) |
| `ENUM('a','b','c')`            | `VARCHAR(n)` with a CHECK constraint, or PostgreSQL `ENUM` type (create type first) |
| `TIMESTAMP DEFAULT CURRENT_TIMESTAMP ON UPDATE CURRENT_TIMESTAMP` | No direct equivalent; use `TIMESTAMP DEFAULT CURRENT_TIMESTAMP` and handle updates in app or with a trigger |
| `MODIFY COLUMN ... ENUM(...)`  | Alter column type or add CHECK constraint |

**Concrete changes:**

1. **AUTO_INCREMENT**  
   Replace with **SERIAL** (or `BIGSERIAL` for large tables):
   - `team_id INT AUTO_INCREMENT PRIMARY KEY` → `team_id SERIAL PRIMARY KEY`
   - Same for `user_id`, `upload_id`, `slab_id`, `deal_id`, `rep_incentive_id`, `team_incentive_id`, `audit_id`, and any legacy `id`.

2. **TINYINT(1)**  
   Use **BOOLEAN** (or SMALLINT 0/1):
   - `is_active TINYINT(1) DEFAULT 1` → `is_active BOOLEAN DEFAULT TRUE`
   - `eligible_for_compensation TINYINT(1) NOT NULL DEFAULT 1` → `eligible_for_compensation BOOLEAN NOT NULL DEFAULT TRUE`
   - In SELECTs, `is_active = 1` → `is_active = TRUE` (or leave as `is_active` for boolean).
   - In Python, `(1 if eligible else 0)` for `eligible_for_compensation` can become `eligible` if the column is BOOLEAN.

3. **ENUM**  
   Two options:
   - **Option A (simplest):** Use `VARCHAR(50)` (or appropriate length) and keep the same values; no CHECK needed for minimal change.
   - **Option B:** Create PostgreSQL ENUM types, e.g.:
     ```sql
     CREATE TYPE user_role AS ENUM ('ADMIN', 'SALES_REP', 'SALES_MANAGER');
     CREATE TYPE upload_status AS ENUM ('DRAFT', 'VALIDATED', 'FINALIZED');
     CREATE TYPE payment_status AS ENUM ('PAID', 'UNPAID', 'PARTIALLY_PAID', 'N/A');
     ```
     Then use e.g. `role user_role NOT NULL DEFAULT 'SALES_REP'`.

4. **ON UPDATE CURRENT_TIMESTAMP**  
   PostgreSQL has no direct equivalent. Either:
   - Drop it and use only `DEFAULT CURRENT_TIMESTAMP` for `created_at` / `updated_at`, or
   - Add a trigger that sets `updated_at = CURRENT_TIMESTAMP` on UPDATE.

5. **ALTER TABLE ... MODIFY COLUMN ... ENUM(...)**  
   In `initialize_schema()` you have:
   ```sql
   ALTER TABLE rep_incentives MODIFY COLUMN payment_status ENUM('PAID','UNPAID','PARTIALLY_PAID','N/A') DEFAULT 'UNPAID'
   ```
   In PostgreSQL:
   - If you use VARCHAR: `ALTER TABLE rep_incentives ALTER COLUMN payment_status TYPE VARCHAR(50);` (and optionally set default).
   - If you use an ENUM type: create the type first, then `ALTER COLUMN payment_status TYPE payment_status_enum USING payment_status::payment_status_enum;`.

6. **Duplicate column / error handling**  
   In migration blocks that run ALTER TABLE and catch “Duplicate column”, also allow PostgreSQL’s “already exists” (e.g. `"already exists" in str(e).lower()` or similar) so reruns are safe.

## 5. Data Migration (Optional)

If you have existing data in MySQL:

1. Export from MySQL (e.g. `mysqldump` with appropriate options, or export tables to CSV).
2. Create the schema in PostgreSQL (run your updated `initialize_schema()` or equivalent).
3. Import data (pg loader, COPY, or custom scripts). Adjust types (e.g. TINYINT → BOOLEAN, ENUM → VARCHAR/ENUM) as needed.
